# ukn设计初衷以及特性




# 简介

`ukn` 是一个基于消息和服务的分布式服务框架，采用`go`编写，致力于简化开发难度和成本


# 设计概述

工作中开发一个分布式多人在线游戏的时候构建了一个`cobweb`的分布式通讯框架（基于`golang`,`c`）。

但是在实际开发过程中代码难以维护以及更新，主要是每次都需要跨平台进行编译，特别是`cgo` 往往需要指定平台的系统库。

而且一些不规范的使用方式造成无法充分发挥多核的优势，可以参见 `关于Go协程的思考` 虽然1.16 支持抢占式，但错误的使用方式依然造成了cpu过高的问题。

再者，高并发服务器会造成过多的 `goroutine` 被创建，极大浪费了内存以及cpu。

基于以上目的，重写了`cobweb` 的底层。 所以 `ukn` 被设计了出来。

虽然与云风大神的`ukn` 同名，但调度方式以及绝大部分api的使用方式是有很大区别的,并且不包含任何lua库或文件。





# 特性

  1. 支持 `cgo` 沿用自 `cobweb`
  2. 支持 `plugins`，为了方便从外部直接加载以满足热更新需求 `unix only`
  3. 支持纯`go`,当然这是一句废话
  4. 支持 `lua` 原因在于 `lua` 是一门简单并且性能很高的脚本语言，能显著降低开发成本
  5. mysql支持，dns支持，http支持
  6. 提供`tcp` `udp`,`unix`支持

# 区别（后续完善）

  1. 集群通讯，`ukn.send`,`ukn.call` 抹平了本地和集群的区别，不存在 `cluster` 这个命名空间或者库,参见 [SPCD](/post/ukn/scpd)
  2. lua 序列化，编码格式完全区别与 `lua-seri.c`。
  3. spb 当前支持 `go` `csharp` `lua`,测试性能高于 protobuf 15%左右，参见 [SPB](/post/ukn/spb)
  4. 纯go实现的http框架，区别于 `net/http`,仅仅实现部分 `RFC` 标准
  5. 支持同步调用超时
  6. 内置消息订阅，也支持 `nats` （一个简单的封装）并不保证线上项目的稳定性，仅仅是一个实验模块
  7. 消息发送默认都是指针，如有需要，可通过一些api来copy，并提供 `ukn.free` 方法来释放它。


# SCPD [点我](/post/ukn/scpd)

> 在`ukn`中每个节点都是平等的，不存在谁依赖于谁，这也是`cobweb`的设计初衷。所以`ukn` 不支持 master/slave 模式，但可以通过业务代码来实现，而非框架本身。



# 编排

`ukn` 是函数式以及低抽象的架构

纯函数式主要是受 `c language` 的影响。也更贴合职责单一的原则

而低抽象是因为`go interface` 并非零成本抽象，它由一定的性能代价。所以整个`ukn` 只有一个接口 `Freeable`。


# 运行

`ukn`执行文件仅仅`6.8mb`,而运行内存仅仅 `2.4mb`，65535个Lua服务仅占用`1.8GB`,也就是每个 `Lua服务` 仅占用 `28kb`,如果是`go`服务那么将会更小
